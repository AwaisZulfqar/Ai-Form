AI Forum — Post Generation, Likes, Comments & Sharing

A lightweight web app that turns a plain-text topic into a fully-formed forum discussion post via OpenAI, then lets users like, comment on, and share it. No auth — a single seeded default user acts for all writes. Scoped for a 4–8 hour build.

01Project Overview

AI Forum turns a topic into a forum discussion post using OpenAI (gpt-4o-mini). Generated posts persist and render in a newest-first feed where users can like, comment, and share. Every post has a dedicated shareable detail page at /posts/:id.

The product deliberately omits authentication, profiles, and moderation. A single seeded Default User is the author/actor for all writes. Emphasis: a clean end-to-end vertical slice — input → AI generation → persistence → feed → interaction.

Visuals follow the "Executive Intelligence Interface" system in stitch_clean_forum_post_generator/ (§8.2): minimal, corporate, content-first — light-gray canvas, white bordered cards, an 8px grid, near-black/slate-navy primary actions.

1.2 Out of scope (do not build)

Authentication/login, profiles, admin panel, notifications, real-time/WebSockets, pagination/infinite scroll, moderation, nested reply threading beyond one visual level, and any multi-user identity system.

02Goals

Product goals

  • G1 — Generate a realistic discussion-starter post from a plain-text topic in one click.
  • G2 — Generated posts persist and appear at the top of a newest-first feed.
  • G3 — A user can like, comment on, and share any post.
  • G4 — Every post is individually addressable via a unique URL (/posts/:id).
  • G5 — Responsive UI with clear loading/error feedback, matching the design system.

Engineering goals

  • EG1 — Clean separation of concerns (routes → controllers → services → models).
  • EG2 — Reusable, composable React components.
  • EG3 — All input validated server-side; secrets never in the client.
  • EG4 — Graceful degradation on AI / DB / network failure with friendly messages.

04Functional Requirements

4.1 AI Post Generation

  • FR-1.1 — Home exposes a topic input + Generate button ("Start a Discussion" card).
  • FR-1.2 — Generate sends POST /api/posts/generate with { topic }.
  • FR-1.3 — Backend validates the topic (§12.1) before any AI call.
  • FR-1.4 — Backend calls OpenAI with a prompt that returns a title + forum-style body (§9.3).
  • FR-1.5 — Parse to { title, content }, validate non-empty (§12.2–12.3), persist a Post by the Default User.
  • FR-1.6 — Return the created post (201).
  • FR-1.7 — Frontend prepends the new post without a full reload.
  • FR-1.8 — During generation, disable the button and show a "Generating post…" skeleton atop the feed.
  • FR-1.9 — Output must read like a discussion starter (opinionated, question-raising), not an encyclopedia paragraph — enforced by the prompt.

4.2 Forum Feed

  • FR-2.1 — Home renders all posts sorted by createdAt descending.
  • FR-2.2 — Each PostCard: title, 3-line preview, author + avatar, relative date, like count, comment count, share action.
  • FR-2.3 — Clicking the title navigates to /posts/:id.
  • FR-2.4 — No posts → render an EmptyState.

4.3 Likes

  • FR-3.1 — Each post shows a like/upvote control with the current count.
  • FR-3.2 — Click sends PATCH /api/posts/:id/like; backend increments likeCount atomically, returns the updated value.
  • FR-3.3 — UI updates optimistically, reconciles with the response.
  • FR-3.4 (optional) — Session guard (state / localStorage Set) blocks a second like in the same session.

4.4 Comments

  • FR-4.1 — Detail page (+ inline card preview) shows all comments, oldest-first.
  • FR-4.2 — CommentForm submits { text }; author defaults to Default User.
  • FR-4.3 — Submit sends POST /api/posts/:id/comments; on 201 append + bump count.
  • FR-4.4 — Each comment: author + avatar, text, relative date.
  • FR-4.5 — No comments → EmptyState inside the section.

4.5 Sharing

  • FR-5.1 — Each post has a share control.
  • FR-5.2 — If navigator.share exists → native sheet with the post URL; else copy https://<host>/posts/:id via navigator.clipboard.writeText.
  • FR-5.3 — Show a toast: "Link copied to clipboard."
  • FR-5.4 — No backend write. PATCH /share is out of scope unless share-count tracking is added.

4.6 Post Detail Page

  • FR-6.1 — /posts/:id fetches post + comments via GET /api/posts/:id.
  • FR-6.2 — Renders full title, content, author, date, like + share buttons, comment list + form.
  • FR-6.3 — Invalid/not-found ID → not-found state (not a crash).

05Non-Functional Requirements

  • NFR-1 Clean code — Separation of concerns; consistent naming; no duplicated logic; UI, business logic, data access isolated.
  • NFR-1a Minimal comments — Write self-documenting code; keep comments sparse. Comment only non-obvious intent (a tricky regex, a business rule, a workaround). No line-by-line narration, no commented-out code, no JSDoc boilerplate on trivial functions. Clear names > comments.
  • NFR-2 Responsive UI — Mobile-first. 12-col desktop (max 1200px), reduced tablet margins, single-column mobile stack (sidebar hidden/collapsed).
  • NFR-3 Loading states — Loader during generation + fetches; buttons disabled in-flight.
  • NFR-4 Error messages — Friendly, no stack traces leaked; retry guidance where relevant.
  • NFR-5 Reusable components — Shared Button, Card, Loader, Input, Avatar, Toast.
  • NFR-6 Environment variables — API key, Mongo URI, port, frontend API base URL via env (§9.5).
  • NFR-7 Security basics — Server-side validation on all inputs; sanitize user text; CORS restricted to the frontend origin; body-size limit; no secrets in the client bundle.
  • NFR-8 Rate limiting (optional/recommended) — Throttle POST /api/posts/generate to protect the AI quota.
  • NFR-9 Accessibility — Semantic HTML; focus states (1px primary border + 2px soft glow); keyboard-usable buttons; avatar alt text.
  • NFR-10 Performance — Feed returns comment counts via denormalized commentCount (§6).

06Database Schema

MongoDB via Mongoose. Three collections. A seed script creates the Default User on first run.

6.1 User

{
  _id: ObjectId,
  name:   { type: String, required: true, trim: true, maxlength: 60 },
  email:  { type: String, required: true, lowercase: true, trim: true },
  avatar: { type: String, default: null },
  createdAt: Date, updatedAt: Date    // timestamps: true
}

Seeded: { name: "Default User", email: "default@aiforum.local", avatar: null }. All writes reference this _id.

6.2 Post

{
  _id: ObjectId,
  title:   { type: String, required: true, trim: true, maxlength: 120 },
  content: { type: String, required: true, trim: true, minlength: 50, maxlength: 2000 },
  author:  { type: ObjectId, ref: "User", required: true },
  topic:   { type: String, trim: true, maxlength: 100 },
  likeCount:    { type: Number, default: 0, min: 0 },
  commentCount: { type: Number, default: 0, min: 0 },   // denormalized for feed
  createdAt: Date, updatedAt: Date
}

Shareable id = the Mongo _id. Share URL is built client-side as ${FRONTEND_BASE_URL}/posts/${_id} — no stored shareUrl. Index { createdAt: -1 } for the feed.

6.3 Comment

{
  _id: ObjectId,
  postId: { type: ObjectId, ref: "Post", required: true, index: true },
  author: { type: ObjectId, ref: "User", required: true },
  text:   { type: String, required: true, trim: true, minlength: 1, maxlength: 500 },
  createdAt: Date, updatedAt: Date
}

On create, $inc the parent post's commentCount in the same flow. Feed/detail read it directly — no aggregation.

Relationships: User (1) ──< (N) Post ──< (N) Comment. User is also the author on comments.

08Frontend Architecture

React 18 + Vite SPA. React Router, two routes. Tailwind configured with §8.2 tokens. Data access isolated in services/ (axios); UI/data state in custom hooks/; ephemeral global state (toast, default user) in context/. No Redux — useState + useContext + custom hooks suffice.

PathPagePurpose
/HomePageTopic form + feed (+ sidebar)
/posts/:idPostDetailPageFull post + comments + actions
*NotFoundPageFallback

State: feed list, generation status, toast in React state/context. On generate success, prepend to the in-memory feed (no refetch). Optimistic like + comment, reconciled with responses.

Spacing, radius, elevation

  • Strict 8px grid. Card padding 16px; vertical gap between posts 24px; icon↔label 4px. 12-col grid, max 1200px, gutter 24px. Mobile single column, 16px margins.
  • Radius: components/buttons/inputs/cards 8px; large containers 16px; chips fully rounded (pill); checkboxes 4px.
  • Elevation — L0 bg #F8FAFC; L1 cards white + 1px border, no shadow; L2 hover 0 4px 12px rgba(15,23,42,.05); L3 modal/toast 0 10px 25px rgba(15,23,42,.10).

Component specs (from design)

  • Navbar — 64px, white, 1px bottom border. Logo left; centered search (decorative/Future); code toggle + avatar right.
  • Post card — white, 1px border, 8px radius. H3 title, 3-line body-sm preview (clamp), footer: avatar + author + · + time + category chip, right actions: upvote (arrow + count), comment (icon + count), share. Hover → L2.
  • Inline comment — 24px left indent per level (max one level); 2px vertical thread line #E2E8F0; body-sm; AUTHOR badge when commenter is the post author.
  • Form input — 1px border, 8px radius, body-md; focus = 1px primary border + 2px soft primary glow (10% opacity); label above.
  • Buttons — Primary: primary bg, white text (black "✦ Generate"). Secondary: white, 1px border, primary text. Heights 40px / 32px.
  • Sidebar (design extra) — "Trending Topics" + "Forum Rules" cards, static/hardcoded, not API-backed. Optional; hide on mobile.
Like vs. Upvote: the design renders the like action as an upvote arrow + count. Keep the API terms (likeCount, /like) but style the control as the upvote.

09Backend Architecture

Layered: routes → validation middleware → controllers → services → models. Controllers are thin; services hold business/AI logic; models are Mongoose schemas. Centralized error middleware maps thrown errors to the envelope.

Request lifecycle

Client → CORS → body parser (size-limited) → rate limiter (generate only)
       → router → validation → controller → service → model → MongoDB
       → controller shapes response → (error?) → errorHandler → JSON envelope → Client

9.3 AI Service — services/aiService.js

  • Single responsibility: given a topic, return { title, content }.
  • Uses the openai SDK with OPENAI_API_KEY, model gpt-4o-mini.
  • Call chat.completions.create with response_format: { type: "json_object" } (JSON mode) → parseable JSON directly.
You are a knowledgeable community member starting a discussion on an online forum.
Write an engaging forum post about the topic: "<TOPIC>".
Requirements:
- A concise, catchy title (max 120 chars), no quotes.
- A body of 80–250 words that raises a question or opinion to spark replies.
- Natural, conversational, discussion-starter tone. Not an encyclopedia entry.
Return ONLY valid JSON, no markdown fences:
{ "title": "...", "content": "..." }
  • Still JSON.parse inside try/catch and strip stray code fences defensively.
  • Post-parse validation: ensure title + content non-empty and within bounds (§12.2–12.3). On failure → throw typed AIError → controller returns 502. Never persist a partial post.
  • Apply a request timeout (~15s) so a hung provider surfaces as a controlled 502.

Middleware

  • validate.js — express-validator chains + formatter → 400 with field errors.
  • errorHandler.js — maps error types → status codes, logs server-side, returns envelope (no stack traces to client).
  • notFound.js — 404 for unknown routes.
  • rateLimiter.js (optional) — express-rate-limit on generate (e.g. 10 req / 15 min / IP).

Seed

seedUser.js idempotently upserts the Default User (boot or npm run seed). Optionally seed 2–3 sample posts so the feed isn't empty on first load.

12Validation Rules

Server-side validation is authoritative; client mirrors it for UX only.

FieldRules
Topic (→ generate)Required; not empty/whitespace; min 3, max 100 chars.
Generated title (post-AI)Required, non-empty; max 120 chars (prefer reject → 502).
Generated content (post-AI)Required, non-empty; min 50, max 2000 chars.
Comment text (→ comments)Required; not empty/whitespace; min 1, max 500 chars.
AuthorAlways the seeded Default User (server-attached). No client author trusted.
IDs / share URL:id validated as ObjectId → 400 if malformed, 404 if absent. Share URL derived only from a valid _id.

13Error Handling

Centralized errorHandler; typed ApiError with statusCode. No stack traces leaked. Standard { success:false, message, errors? } envelope.

ScenarioBackendFrontend
AI failure (down, bad key, rate limit, timeout)502/500, friendly message, do not save, log"Unable to generate post right now. Please try again." + retry; keep topic input
Empty/malformed AI responseReject, controlled 502, no persistSame generate error + retry
DB failure500, log, generic messageGeneric error toast, retry
Network failure—Clear network message, preserve input, retry
Invalid request400 with field errors[]Inline field errors; keep form state
Not found404Not-found state on detail; toast on actions
Rate limited (if enabled)429"You're generating too fast — try again shortly."
Principles: fail closed on writes (never persist invalid data); user copy is friendly and actionable; internal details are logged, not exposed.

16Future Improvements

Outside required scope. Prioritized:

  1. Search discussions — wire the navbar search to a filtered feed query.
  2. Trending topics / category chips — make sidebar + categories data-driven.
  3. Delete comment — with confirm.
  4. Dark mode — tokens already define inverse surfaces; add a toggle.
